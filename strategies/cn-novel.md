# 策略：中文长篇小说 (Chinese Novels - TXT/EPUB)

## 背景与痛点
中文长篇网络小说动辄数百万字。NotebookLM 所谓的 50 万“Word”限制对中文极其不友好，因为 1 个汉字可能消耗 1-2 个 Token，50 万“字”极易突破底层的 100 万 Token 窗口，导致被模型**静默截断（大结局剧情丢失）**。

## 核心策略：正则识别 + Tokuin 计价 + 安全切分

1.  **格式统一**
    *   如果是 EPUB，先用 `pandoc` 等工具提取并转换为平铺的 `.md` 或 `.txt`。
    *   如果是平铺文本，必须确保其具有 Markdown 层级，至少章节标题要加上 `# `。

2.  **定位语义边界**
    *   长篇小说绝对不能从自然段中间切断！
    *   使用正则表达式定位章节标题作为安全的切分点。常见的中文章节正则：
        `^(#\s+)?(第[零一二三四五六七八九十百千万0-9]+[章卷回节折].*)$`

3.  **Token 累加与分块**
    *   写一个脚本，逐章读取内容。
    *   使用 `tokuin` 计算当前累加内容的 Token 数：
        `tokuin <file_or_content> --model gemini-1.5-pro`
    *   不断累加当前块的 Token 数。当当前块的 Token 累计达到 **700,000 - 800,000 Token**（安全线）时：
        *   **停止累加，在此章节**之前**切出一刀。**
        *   将累加好的内容保存为一个 `.md` 文件。
        *   将当前章作为下一个 `.md` 文件的开头继续累加。
    *   *原则：宁可没能尽可能触及到上限导致浪费，也要将截断风险降到可以忽略。*

4.  **有序命名**
    *   生成的文件必须包含序号，如：
        `01_凡人修仙传_第1章-第154章.md`
        `02_凡人修仙传_第155章-第320章.md`

5.  **跨 Notebook 策略**
    *   如果切出了超过 48 个 `.md` 文件，说明小说过长，必须在 NotebookLM 中创建“修仙传_上部”、“修仙传_下部”等多个不同的 Notebook 分别导入。
